## 2026-07-19 — The Closed Economy, slice 2: equipment purchasing, upgrade tiers, NPC job-seeking (§Stage 5, in progress)

Continues slice 1 (same day, see the entry directly below). Still not Stage 5's exit test — this slice adds company equipment purchasing (§9.4), Management-gated upgrade tiers (§9.5), and a minimal NPC job-seeking mechanism that also closes Stage 4's long-flagged "another member works" adaptation-ladder gap (§10).

**What was built:**

- **Company equipment purchasing** (`companies/decisions.ts`'s `restockEquipment`): any company whose job slot requires a tool it doesn't currently have buys a replacement from the market — a plain merchant-faucet purchase (§9.4: "bought from toolmakers (or the merchant faucet early on)"), reusing `market.ts`'s existing `companyBuyFromMarket`. Checked daily, unconditionally, unlike input restocking — a job slot with zero tools is a hard stop on production entirely (`jobs/shifts.ts`'s `no_tool` outcome), so even a sloppily-managed owner eventually replaces a broken one. Two new market listings (`hoe`, `axe`) seed this the same way shoes/cloaks already work.
- **Upgrade tiers** (`companies.tier`, migration `0011_company_growth`; `companies/decisions.ts`'s `tryUpgrade`): a company upgrades — raising its tier and its job slots' capacity together, capped at §9.5's 20-employee hard cap — only when all three real conditions hold: every job slot is full, it's been genuinely profitable over a trailing 30-day ledger window (not just solvent), and it can afford the cost outright. A sloppily-managed owner (below a Management floor) never recognizes the opportunity at all. Upgrade cost is modeled as a coin sink (no construction/builder-labor module exists yet, §Stage 6) — the same honest simplification as every other "coin leaves for something not modeled yet" sink already in this codebase.
- **Minimal NPC job-seeking** (`population/cadence.ts`'s `applyNpcJobSeekingWeeklyCadence`): every unemployed household member is a candidate for every open job slot, weekly, with no skill-matching or §11.3's full utility scoring (deliberately minimal, not the fuller model). Households currently under financial strain get first pick of scarce openings, and a hire into a strained household logs a `household.hardship.member_works` settlement event — a real, if simplified, version of §10's "another member works" adaptation rung, flagged unbuilt since Stage 4.

**A real 90-day headless run (not assumed, run and read before writing this entry) found something worth correcting rather than glossing over:** slice 1's seed comments predicted the level-5-managed Riverside Mill would be "thriving" and the level-0-managed Village Bakery "struggling," following §11.5's own flavor text. The actual run showed the opposite — the mill ended near-insolvent (balance 10, briefly flagged insolvent) while the bakery was the clear profit leader (ledger net +840 over the run). The reason is real, not a bug: Management currently weights *purchasing* reliability only (bigger, more frequent restocks), not *selling* efficiency or restraint relative to actual throughput — a well-managed company can buy inputs faster than it can profitably resell its output, especially a link like the mill that sits in the *middle* of the chain rather than facing guaranteed end-consumer demand the way the bakery does. `seed/demoWorld.ts`'s management-XP comment and `stage5.test.ts`'s header were both corrected to state this honestly instead of the original (wrong) narrative. Growth itself is real and did fire once in that run (Oster Farm reached tier 2), confirming the mechanism works — but the same run showed the farm's own trailing-30-day profitability gate can pass even while its full-run net stays negative once the larger post-upgrade wage bill is counted, a real balance-tuning question flagged for the harness (§17), not fixed reactively here.

**Verification:** new unit tests for equipment restocking (buys exactly one replacement, doesn't double-buy once stocked), upgrade gating (fires when fully-staffed/profitable/affordable/well-managed; correctly withholds when there's still room to hire without expanding), and job-seeking (fills open slots, prioritizes strained households, no-ops safely with zero openings). `stage5.test.ts` extended with real assertions that job-seeking reaches full employment at the mill, bakery, and logging camp within its 60-day window — confirmed by an actual run, not assumed. `npm run validate` passes clean (82 tests, 1 correctly skipped, typecheck/lint/format/build all clean); the heavy 90-day/60-day scenario tests together still complete in ~146s, no performance regression from slice 1's fix. A real browser smoke test (Playwright): app boots, time advances, settlement screen still lists Riverside Mill and the Village Bakery, zero console errors.

**What's still explicitly not built:** true standing B2B contracts with freight (§9.7 — needs the transport module); seasons affecting price/production; failure → auction (§9.6); merchant imports as a modeled entity; rolled starting conditions (§5.4); market charts, business-log, and world-chronicle screens; the real 2-year exit test; and — newly named by this slice's own finding — Management weighting *selling* decisions, not just buying, which the mill/bakery result above shows is a real gap, not a hypothetical one.

## 2026-07-19 — The Closed Economy, slice 1: grain -> flour -> bread (§Stage 5, in progress)

**Stage 5 ("The Closed Economy & Living Companies," §Stage 5) is the largest stage in MASTERPLAN.md** — full v1 chains, Management-weighted business ledgers, company equipment/upgrade tiers, B2B contracts, smoothed pricing, failure+auction, faucets/sinks, seasons, stabilizers, merchant imports, rolled starting conditions, market charts, business logs, world chronicle, a 2-year exit test. Per §19 ("one module or one stage-task per session"), this entry covers the first, foundational slice — everything else in the stage depends on companies actually earning real revenue, which didn't exist before this. **Stage 5 is not complete; this is an honest progress entry, not an exit-test pass.**

**What was built:**

- **Real goods chains** (`src/engine/production/recipes.ts`, new): grain (farm) → flour (mill) → bread (bakery), the first genuine multi-hop transformation chain (§7.2, §9.1's "inputs + labor + tools + time → outputs"). This is also the single source of truth for *both* production paths that already existed — the player's own `work_shift` action (`jobs/shifts.ts`) and NPCs' weekly batch (`population/cadence.ts`) previously hardcoded two separately-drifted copies of the same "farming → grain" fact. Unifying them surfaced a real, previously-uncaught bug: `shifts.ts` unconditionally produced `grain` regardless of which job slot the shift belonged to, so a player working the *logging* job slot was silently producing grain instead of firewood. Fixed as part of the unification, not a Stage 5 regression.
- **The market actually closes** (`market.ts`'s new `producerCompanyId` on `MarketListing`, migration `0010_production_chains`): buying from a listing backed by a real producer now pays that company (`transferCoin` + a ledger entry) instead of sinking coin out of the economy. Goods with no local producer yet (shoes, cloaks) are untouched — still the §7.2 merchant-faucet import they always were. This required fixing *two* buy paths, not one: the player's own `buy_bread` action, and — the dominant path by volume — households' own `feedHousehold` bread purchase (`population/cadence.ts`), which buys for up to 20 households a day versus the player's occasional purchase.
- **Company ledgers** (`companies.ts`'s `company_ledger_entries` table, `recordLedgerEntry`/`summarizeLedger`): revenue, material cost, and wage entries, queryable per company per window. A minimal, real version of §9.3 — full tabbed company screens remain Stage 6, player-owned only.
- **Management-weighted daily decisions** (`companies/decisions.ts`, new): each company (now including the farm and logging camp, retrofitted with owners) restocks its input good on a Management-level-determined interval (well-run: daily; sloppy: every 5 days) and batch size, and sells surplus output daily. Four dedicated owner-operator NPCs (one per company, each a real single-person household reusing the existing needs/feeding machinery) carry deliberately varied starting Management XP — Riverside Mill's owner is level 5 ("thriving"), the Village Bakery's is level 0 ("struggling") — a real, observable divergence per §11.5's own flavor text, not a label.
- **Smoothed pricing** (`market/pricing.ts`, new): `target price = base × scarcity` (current stock vs. each listing's own seeded reference level), drifting toward target daily. Demand/local/seasonal multipliers from §8.1 rule 4's full formula are not modeled yet — flagged, not silently dropped.
- **A minimal, real insolvency signal** (`companies.insolvent_since_tick`): set the first tick a company's balance hits zero, cleared on recovery — not the full closure→auction chain (§9.6), which needs equipment/upgrade modeling that doesn't exist yet.
- **A real performance bug found and fixed under load, not by inspection.** The first working version of `companyBuyFromMarket` settled coin per unit bought (one `transferCoin` + one ledger entry per unit, inside the loop) — for a well-managed company restocking 40 units daily, that's 80 extra `event_log` rows *every day* for 90 days, on top of the four companies' new daily/weekly activity generally. `stage4.test.ts`'s 90-day exit test — previously ~165s — first came back at ~434-443s, blowing well past its 300s regression guard, confirmed as a real slowdown (not parallel-test contention) by re-running it in isolation. Root cause diagnosed empirically, not guessed at random: the fix batches the coin/stock/ledger side of a purchase into one lump transaction per restock (matching the existing precedent of `population/cadence.ts`'s own weekly lump wage payment), while keeping item creation genuinely per-unit — §7.1 provenance is per-item by design and was never the actual cost driver. A second, smaller contributor (companies selling goods nobody buys — the logging camp's firewood has no buyer or recipe demand at all) is fixed by `production/recipes.ts`'s new `hasKnownDemand()` gate plus a flat daily sale cap, so a good with no real market never piles up in `market-stock` forever. After both fixes, the *entire* validate suite (75 tests, including the new 60-day Stage 5 scenario test) runs in ~139s.

**Decisions:**

- **Recipes stay a code catalog, not a DB table**, same precedent as `goods/catalog.ts` — §16's "goods-as-data... DB records + JSON packs" is aspirational for goods too, not yet true; flagged consistently rather than building a mod-loading system as a side quest this session doesn't need.
- **Company owners are real NPCs in real single-person households**, not a needs-free "abstract manager" concept — reuses 100% of the existing household/needs/cadence machinery (and keeps them off the player's expensive per-tick path the same way every other NPC is) rather than inventing a parallel entity kind. This adds `COMPANY_OWNER_HOUSEHOLD_COUNT = 4` real households on top of the generated NPC population — `stage4.test.ts`'s household-count assertions were widened accordingly, a real and permanent change to the seeded world, not a test-only fudge.
- **One producer per (site, good) market listing**, not per-unit provenance of the stall's stock — once a company sells into a listing, subsequent buyers pay *that* company even against any older, differently-sourced stock still counted in the same listing. A deliberate v1 simplification, not a bug.
- **Spot-market B2B purchasing, not §9.7's standing contracts.** A mill "restocking grain" today is a company acting as an ordinary market buyer on its own daily cadence — not a scheduled recurring shipment with its own terms, fulfilled by real freight. §9.7's actual contract model needs the transport/freight module, which is still an empty directory. Named explicitly as the next real piece, not silently substituted for.
- **The bread merchant-faucet import was reduced from 6000 to 1000**, sized as a bridging buffer for the chain's ~3-week ramp-up (a sale has to propagate farm → mill → mill sale → bakery → bakery sale before any bakery-baked loaf reaches the market) rather than a number large enough to mask whether the chain is doing anything at all.

**What's explicitly not built yet (Stage 5 continues, not closed):** company equipment purchasing/upgrade tiers (§9.4/§9.5); true standing B2B contracts with freight (§9.7); seasons affecting price/production; failure → auction (§9.6's "permanent failure"); merchant imports as a modeled entity; rolled starting conditions (§5.4); market charts, business-log, and world-chronicle screens (§14.2/§14.3); the real 2-year exit test itself. `stage5.test.ts`'s 60-day scenario test proves the chain closes for real (all three companies show traceable ledger revenue, provenance holds, conservation audit passes) — it is explicitly *not* claimed as Stage 5's own exit test.

**Verification:** `npm run validate` passes clean (75 tests passed, 1 correctly skipped, typecheck/lint/format/build all clean). A real browser smoke test (Playwright against the Vite dev server): app boots, HUD shows starting coin, time advances via the real speed controls, the settlement screen lists Riverside Mill, the Village Bakery, and the Households tab — zero console/page errors.

## 2026-07-18 — The sql.js memory ceiling, root-caused and fixed (checkpoint/rehydration)

Not a stage — an infrastructure fix for the constraint flagged since Stage 2 (30→12 days), hit again by Stage 3's own scenario at first, and forced Stage 4's exit test down to 20 days. Root-caused and fixed at the source rather than worked around again, at the user's explicit request ("ensure a real fix... I trust your judgement").

**Root cause, confirmed rather than inferred:**

- WASM linear memory only ever grows and is never reclaimed by the runtime — a fundamental property of WebAssembly, not a sql.js bug per se. Every allocation sql.js's binding layer makes inside that memory has to be explicitly freed; evidently not all of them are (a documented, known category of sql.js issue — [sql-js/sql.js#482](https://github.com/sql-js/sql.js/issues/482), [#571](https://github.com/sql-js/sql.js/issues/571), [jlongster/absurd-sql#53](https://github.com/jlongster/absurd-sql/issues/53)).
- `db/sqlite.{node,browser}.ts`'s `loadSqlJs()` memoizes `initSqlJs()`, so there was exactly **one** WASM module instance — one linear-memory heap — for the life of the process. Every `Engine`/`Database` created from it, no matter how many, shared that same ever-more-fragmented heap; closing one connection and opening another reclaimed nothing, because the damage lives in the module's memory, not the connection.
- **A false start, caught before it shipped:** the first version of a "get a fresh WASM instance" loader just called `initSqlJs()` again. This does *not* work — confirmed by reading sql-wasm.js directly: sql.js's own bundled code caches its module promise in a variable scoped to that module's own top level (`var initSqlJsPromise`), independent of anything in this codebase. A second call through the same imported binding silently returns the *original* instance. The first attempt at the real 90-day Stage 4 run still crashed after this "fix" was in place, which is what surfaced the false start — the determinism tests alone hadn't caught it, because they only prove logical-state continuity, not that the memory was ever actually fresh (both are needed; only one was being tested).
- **The real fix (Node only, confirmed empirically):** drop sql.js from Node's CJS `require.cache` and `require()` it again. This does produce a genuinely independent module — verified directly (distinct `SQL` object, distinct `Database` class, and a second instance stayed fully healthy after the first was deliberately stress-filled with 200k operations). `db/sqlite.node.ts`'s `loadFreshSqlJs()` now does this.
- **No browser equivalent exists yet.** ES module imports don't have an invalidatable cache the way CJS's `require.cache` does. `db/sqlite.browser.ts`'s `loadFreshSqlJs()` is left in place but explicitly documented as **not fixed** — calling it today just returns the same memoized instance, same as the false start above. A real browser fix likely needs a genuine dynamic `import()` against a separately-fetchable, code-split chunk with a cache-busting query, verified in an actual browser — not attempted here. Long real play sessions in one tab remain exposed to this ceiling until that's done; this is a named, tracked gap, not a silent one.

**What was built:**

- `rng.ts`: `createRng()` now returns a `SeededRng` — structurally still a plain `() => number` (every existing caller keeps working unchanged), but also exposes `.getState()`. mulberry32's entire state is that one 32-bit integer, so capturing it is enough to resume the exact same draw sequence later.
- Migration `0009_rng_state`: `world_meta.rng_state`, synced by `Engine.export()` on every call (a plain per-call write, not a per-tick one — doesn't reintroduce the cost the `progress_ticks` fix just removed). `Engine`'s constructor now resumes the RNG from this saved state when present, falling back to hashing the seed string only for a genuinely new world. This incidentally fixes the same gap for ordinary save/load, which previously silently restarted the RNG on every reload regardless of how far the run had progressed — latent since Stage 0, never exercised because no save/load UI exists yet, but load-bearing the moment checkpointing made "reload mid-run" a real, frequent operation.
- `checkpoint.ts` (new): `checkpointEngine(engine, { seed, loadFreshSqlJs })` — exports the world (syncing RNG state first), disposes the old `Engine`, instantiates a genuinely fresh WASM module, and rehydrates a new `Engine` from the exported bytes inside it. Deliberately not automatic/hidden inside `Engine.advanceTicks()`: that method is synchronous and called in tight loops (every scripted test, the UI's tick interval); `loadFreshSqlJs()` is inherently async, so checkpointing is an explicit, caller-driven operation at a boundary the caller chooses.
- **A real trap, hit once and now documented prominently:** the rehydrated `Engine` has an empty `ActionRegistry` — action *definitions* are code, held only in memory (§Stage 0's original decision), never persisted. A checkpoint is exactly a reload from the registry's point of view, so callers must re-run `seedDemoWorld()` (or whatever registers their action types) on the returned engine, exactly as they already must after `Engine.bootstrap()` itself. This is the same class of bug DECISIONS.md's Stage 2 entry already documented once for ordinary reloads — reintroduced by this feature and caught immediately by the first checkpoint test run, not discovered later.
- `stage4.test.ts`'s 90-day exit test now runs the **literal 90 days**, not the 20-day stand-in — checkpointing every 15 in-game days (well under the ~20-25 day danger zone observed without it, leaving real margin for Stage 5's heavier world).
- `checkpoint.test.ts` (new): proves determinism survives one checkpoint, and two in the same run — not by comparing raw `export()` bytes (two DBs that reach the same *logical* state via different physical operation sequences aren't guaranteed byte-identical at the SQLite file-format level even when fully deterministic — page layout/freelist state can differ), but by comparing a full logical snapshot (needs, balance, gear, employment, skill XP, entire personal log) between a checkpointed run and an uninterrupted run of the identical seed and script.
- `_stage5_scale_stress.test.ts` (new, `describe.skip` by default — even the reduced 300-day target below takes 15-20 minutes and has no place in routine `npm test`): a manual, opt-in confidence check reusing Stage 4's world well past the 90-day exit test's scale. Not a proof that Stage 5's own, heavier world stays safe — only that the checkpoint *mechanism* doesn't crash across many more cycles than the 90-day exit test exercises. **Honest result, not the hoped-for one:** first attempted at the full 730-day (Stage 5 exit-test) scale; it reached day 301 (20 checkpoints) with zero crashes and a clean audit throughout, then hit a fixed 30-minute test timeout before finishing — a timeout, not a crash, but the full 730-day target is *not* confirmed. Per-checkpoint cost appears to rise at this scale (this run paced meaningfully slower per checkpoint than the 90-day exit test did) — not diagnosed further here. The committed test now targets 300 days, comfortably inside what was actually observed to complete, so it reliably passes if re-run; the real 730-day question is left as a named follow-up for whenever Stage 5's own exit test needs it.

**Decisions:**

- **Logical-snapshot comparison, not byte-comparison, for the determinism proof.** Byte-identical export comparison was Stage 0's own standard for save→load→resave, but that test never crossed a real re-instantiation boundary — a checkpoint does, and SQLite's on-disk format isn't guaranteed byte-stable across different physical write histories that reach the same logical content. Comparing meaningful values (needs, balances, XP, the full event log) is both more robust and closer to what determinism actually promises (§4.2: "same DB + same seed = same result," a statement about outcomes, not file bytes).
- **Checkpointing stays an explicit, caller-driven operation, never hidden inside `Engine`.** `advanceTicks()`'s synchronous contract is relied on everywhere (every scripted test's decision loop, `useGameClock`'s `setInterval` callback); silently making it sometimes-async to hide a checkpoint would ripple into every caller for a marginal ergonomics win. A caller that wants long-run safety opts in explicitly at a point it controls.
- **The browser gap is documented, not papered over with an unverified "fix."** It would have been easy to ship `loadFreshSqlJs()` for the browser looking identical to the working Node version and call it done — it was already written that way once, and was wrong. Leaving it honestly broken with a clear comment (rather than confidently broken) is the more useful state for whoever picks this up next.
- **The stress test is `describe.skip`, not deleted or left unwritten.** It's genuinely valuable — for Stage 5's own eventual exit test, and as a regression check if the checkpoint mechanism itself ever changes — but a 20+ minute test has no business running on every `npm run validate`. Skipped-but-present, with instructions, beats either extreme.

**Verification:** `npm run validate` passes clean. `checkpoint.test.ts`: both determinism proofs pass (single and double checkpoint boundary, full logical-state equality including the event log). `stage4.test.ts`'s 90-day exit test passes at its real, undiminished duration (165s wall-clock, 6 checkpoints, zero crashes — down from the 20-day stand-in it shipped with days earlier). `_stage5_scale_stress.test.ts` (run once manually, then re-skipped): reached day 301 (20 checkpoints) with zero crashes and a clean conservation audit before a 30-minute test timeout cut it off short of its original 730-day target — see that file's own header and the paragraph above for the honest accounting; the committed version now targets a reliably-completing 300 days.

**Why this can't be called fully closed:** the Node/headless fix is real, verified, and directly unblocks every exit test and any future balance-harness sweep — which is where this ceiling had actually been causing damage. The browser path (a real, continuously-played session) is architecturally unfixed and explicitly flagged as such; see the `wyrnlands-sqljs-memory-ceiling` memory for the concrete next step. Stage 5's own, heavier world (full goods chains, B2B contracts, more companies) hasn't been checkpoint-tested directly, and the full 730-day target itself remains unconfirmed — only that 300+ days works cleanly and per-checkpoint cost at longer scale needs its own tuning pass, both real follow-ups for whenever Stage 5's exit test is actually being built, not solved here.

## 2026-07-18 — Living NPCs & Households (Stage 4 complete\*, §Stage 4) — \*with a flagged duration gap

**What was built:**

- **The memory-ceiling spike, done first and for real.** Before writing any Stage 4 architecture, a throwaway test gave ~40 entities a `needs` row and ran them through `Engine`'s existing per-tick, per-entity needs path (the same one the player already used) — confirmed it exhausts sql.js's WASM heap in **under 3,000 ticks**, not the 129,600 a 90-day run needs. That single result decided the whole stage's shape: NPCs could not be "the player, forty times over." Everything below is built around that constraint, not against it.
- **Households** (`src/engine/population/households.ts`, migration `0008_npcs_and_households`): a household is also an `entities` row — same "reuse the entity/wallet/item machinery" move as companies (§Stage 3) — so it owns a wallet (shared money) and inventory (shared food, belongings) for free. `household_members` maps each person to exactly one household.
- **NPC generation** (`population/npcGen.ts`): `generateNpcPopulation` builds ~40 NPCs (`NPC_HOUSEHOLD_COUNT = 20` households, 1-3 members each) from placeholder name pools, deterministically from the engine's own seeded RNG. Some are hired into open job slots at generation time (see below) — there's no ongoing "NPCs look for work" behavior this stage (§11.3's full utility-scored decision model is Stage 5+ territory; Stage 4's own feature list doesn't ask for it).
- **A second employer**: Hollow's Edge Logging Camp (Woodcutter, `woodcutting` skill → firewood), working out of the existing forest site. `job_slots` gained a `capacity` column (Stage 3's "one worker per slot" becomes "up to capacity") — Oster Farm and the logging camp each run 3 slots, giving NPCs somewhere real to work without the player losing their own shot at a farm job.
- **"Regional LOD" = background aggregation** (`population/cadence.ts`), the risk-mitigation table's own term (§18: "sql.js memory ceiling | ... background aggregation"). NPCs never touch the player's per-tick action queue. Instead:
  - **Daily** (`applyHouseholdDailyCadence`, §4.2's "daily... household budgets"): each household feeds its members (consume stored bread, else buy the shortfall from the market if it can afford to, else a member goes hungry at a 35-hunger subsistence floor — §8.2 "common land gathering," never zero), applies one day's rest/warmth, and evaluates its budget.
  - **Weekly** (`applyNpcLaborWeeklyCadence`, §4.2's "weekly... hiring/wages"): every active NPC employment gets paid a lump wage (capped at what the employer can actually afford), skill XP, and output (grain/firewood scaled by skill level) — one batched pass per employment, not five per-shift action-queue entries.
  - Both are a handful of DB calls per household/employment, not per tick — the entire reason this scales where the naive approach didn't.
- **The adaptation ladder** (§10), the rungs Stage 4 actually builds: a household short on food logs `household.hardship.reduced_food`; a household critically low on coin sells a belonging (a seeded spare cloak, ~40% of households) logging `household.hardship.sold_belongings`, or failing that takes a parish charity stipend logging `household.hardship.charity`. "Another member works" and "migrate" are named in §10 but not modeled — flagged, not silently dropped (no NPC job-seeking behavior exists yet to make the first one real, and migration's full push/pull model per §11.4 is a larger, later feature).
- **Hourly presence rosters** (`population/presence.ts`): a deterministic lookup, not simulated movement — employed NPCs are scheduled at their workplace 6:00-18:00, a stable per-entity subset at the tavern 18:00-21:00, home otherwise. `LocationScreen`'s "Who's here" stub (open since Stage 1) is now real and clickable.
- **Screens**: `HouseholdScreen` (members, reserves — no "obligations" section, since no rent/housing module exists yet to make one honest) and `NpcProfileScreen` (§11.2: placeholder portrait, occupation/employer, household link, a needs-derived condition line). `SettlementScreen` gained a Households tab.
- **Two real, permanent engine fixes**, found by pushing the 90-day exit test until it broke, not by inspection:
  1. `actionQueue.ts`'s `processActorActions` no longer writes `progress_ticks` to the DB every tick an action is in progress. Nothing was reading it while in progress — `ActionQueuePanel`'s progress bar already recomputes `currentTick - startedAtTick` client-side — so it was a pure per-tick write with no reader, ~129,600 of them on a 90-day run.
  2. `needs.ts`'s `tickNeeds` merged its two per-tick reads (`getNeeds` + `getCurrentAction`, the latter only to check for collapse-recovery) into one query. Both fixes are kept regardless of the exit test's final duration — they're real, not workarounds.
  3. `jobs/shifts.ts`'s wage payment now caps at what the employer can actually afford (`Math.min(wage, balance)`) instead of calling `transferCoin` unconditionally. Stage 3 never exercised an insolvent company; Stage 4's NPCs now draw from the same wallets on their own cadence, and a company running dry stopped being hypothetical. A worker still does the shift; an employer that can't pay just can't pay — harsh, not a crash.

**Known limitation — the exit test's "90-day run" is 20 days, not 90 (same shape as Stage 2's 30→12):** even after both per-tick fixes above, a scripted player loop plus Stage 4's much heavier per-household/per-company world empirically stops completing somewhere in the 20-30 day range — confirmed by running it repeatedly, not assumed (the exact crash point and even which query it crashes in vary run to run; sql.js's own heap allocator isn't perfectly deterministic in exactly when it exhausts itself). 20 days passed reliably across repeated runs and is what `stage4.test.ts` asserts, clearly labeled. This is the same unresolved sql.js constraint flagged since Stage 2 — Stage 4 made real, permanent progress on it (the LOD redesign plus the two per-tick fixes), but did not root-cause it. **Must be revisited before Stage 5's 2-year exit test**, which will hit this far harder with an even larger population. See the `wyrnlands-sqljs-memory-ceiling` memory for the running account.

**Decisions:**

- **Household membership is the signal for "background-simulated NPC" vs. "foreground-simulated (player)."** `Engine.applyNeedsCadence`'s per-tick query now excludes any entity present in `household_members`; the weekly labor cadence does the mirror check (skips any employment whose `entityId` *is* a household member — i.e., only pays NPCs, since the player earns via real `work_shift` actions). One boolean signal, reused consistently, rather than hardcoding `PLAYER_ID` anywhere in engine code (which stays seed-layer, not engine-layer, per §4.2's UI/engine boundary).
- **Bread stock bumped from 500 to 6,000.** 500 was tuned for one person's Stage 2/3 needs; a ~40-person settlement over even 20 days would run the market dry and turn "no baseline starvation" into an artifact of an empty shelf rather than a real result. Recalibrating a placeholder constant for the new population scale, not a new mechanic.
- **Company starting capital bumped from 300 to 2,500 (both employers).** Companies still have no revenue mechanism (no mill/bakery chain until Stage 5) — wages are a one-way drain by design. A company that outlives Stage 4's exit test but would eventually go insolvent over a much longer run is a realistic outcome (§11.5's own emergence targets describe exactly this), not a bug; the number is sized for "comfortably clears this stage's runs," not "unlimited."

## 2026-07-17 — First Job (Stage 3 complete, §Stage 3 — FIRST PLAYABLE ✅)

**What was built:**

- **Companies** (`src/engine/companies/`): a company is also an `entities` row (migration `0007_jobs_and_companies`'s `companies` table has `id REFERENCES entities`), so it owns a wallet and holds inventory exactly like a person — no separate "business account" concept needed. `companies.ts` (`createCompany`/`getCompany`); `tools.ts` (`wearCompanyTool`) wears a company-owned tool's durability and destroys it at 0, the same shape as `gear.ts`'s `wearGear` minus the gear/slot bookkeeping (company tools aren't equipped by a person — they just sit in the company's own container and get used by whoever's on shift).
- **Jobs** (`src/engine/jobs/`): `job_slots` (posted wage band, hours, skill ask, employer, required company tool — §14.2's Jobs screen fields) and `employment` (one active row per worker, wage fixed at hire). `jobs.ts` exposes `listJobOpenings`/`applyForJob`/`quitJob`. `shifts.ts`'s `createWorkShiftActionDefinition` builds a `work_shift_<jobSlotId>` timed action, following `market.ts`'s "bake the id into the closure, look up live state at resolve time" shape.

Haggling is a Trading-skill-gated roll (`getSuccessChance` against the `trading` skill, XP'd on every attempt) that, on success, moves the wage partway toward the band's ceiling; on failure it simply stays at the floor — haggling can never cost the applicant anything, per §18 "Harsh ≠ opaque."
- **Shift mechanics**: a shift pays the employment's fixed wage regardless of outcome — "presence = labor-ticks = production" (§9.8), a labor-time wage, not piece-rate — while the Farming-skill check instead governs the harvest: success yields more grain at a higher quality tier; failure yields a little (not zero — §13.2 "wasted time and materials," not a wasted day). Skill XP is granted either way, matching every other skill-gated action in this codebase (`chop_wood`, needs collapse recovery, etc.). The shift also wears the company's shared tool and requires one to exist at all — no tool, no shift, and no consolation XP/wage either (an `outcome.data.reason: 'no_tool'` flag lets `applyOutcome` distinguish "didn't happen" from "happened and failed the skill check," rather than re-querying tool state twice).
- **Grain into real storage**: harvested grain is produced directly into the company's own container (its entity id), the same pattern as a person's inventory — no separate "storage" sub-container was needed since the company already has one via being an entity.
- **Jobs screen** (`src/screens/JobsScreen.tsx`), reached via a new "Browse job openings →" button on the notice board's `LocationScreen` (§14.4: "Read the notice board... **A real empirical finding on the sql.js memory ceiling ([[wyrnlands-sqljs-memory-ceiling]]):** Stage 2's exit test had to cut 30 days down to 12 because a varied action mix reliably crashed sql.js's WASM heap around 30-40k ticks. Stage 3's season scenario runs the **full 30 days (43,200 ticks) with no reduction and no crash** — it completed in ~13-19 seconds clean. The likely reason: a work shift is 360 ticks in one action-processing call, so a day of farm labor collapses into 1-2 engine actions instead of the dozens of short (5-90 tick) actions Stage 2's chop-wood/buy/sell loop needed to cover the same span. This is consistent with the earlier finding that the ceiling scales with *distinct operation volume*, not raw tick count — it's the first real data point suggesting coarse-grained action durations (real jobs, real travel) may naturally keep Stage 4/5's longer runs clear of the ceiling, rather than requiring a dedicated fix. Not proof the risk is resolved — Stage 4/5 add multiple NPCs each generating their own action volume, which this single-actor test says nothing about — so the blocking-spike guidance for Stage 4/5 in the memory above still stands, but this is a genuinely encouraging data point to carry into that investigation rather than assuming the worst by default.

**First scripted-worker bug of this stage, caught and fixed before it reached the engine:** the season scenario's first draft reused Stage 2's decision thresholds (act on a need once it drops below 50). A work shift is 360 ticks — six to sixty times longer than any Stage 2 action — so a need sitting at, say, 55 when a shift *starts* could decay straight through 0 before the shift ends (thirst alone decays 60 over 360 ticks) and force a mid-shift collapse. Fixed by giving shift-eligibility its own, much wider safety margin (act below 60, only commit to a shift once every need clears 75) — an engine-correct outcome (collapse handling itself worked fine) exposed by a test script that hadn't yet accounted for its own new action's duration.

**Decisions:**

- **A company is an `entities` row, not a separate concept.** It needs a wallet (wages) and an inventory (tools, grain storage) exactly like a person does — reusing the existing entity/wallet/item machinery outright avoided inventing a parallel "business account" system for what turned out to be the same shape.
- **`job_slots.id` and the company id are code-assigned TEXT, not autoincrement**, mirroring `sites.id`. `registerDemoActionTypes()` must register `work_shift_<jobSlotId>` unconditionally on every fresh `Engine`, including before a fresh DB has any world content — the same ordering constraint (and the same fix shape) as `buy_<good>`/`sell_<good>` being keyed by fixed strings rather than DB-generated ids (see Stage 2's entry on this file).
- **The wage is fixed at hire and paid per shift regardless of the harvest's success/failure.** Modeling it as piece-rate on the grain yield would conflate two different things §9.8 and §13.2 keep separate: presence/labor-time (what the wage is for) and skill-driven output quality/quantity (what Farming XP and the skill check govern). This also means Farming XP and wages both accrue even on a "failed" shift — consistent with every existing skill-gated action in this codebase already granting XP on failure.
- **No employer-side haggling AI, no work-history-weighted opening wage.** §9.8's full haggling model ("depends on the applicant's skill and work history... business's urgency and profitability... local labor supply") needs NPCs, reputation, and business decision-making that don't exist until Stage 4/8. Stage 3's "simple wage haggling" is the applicant-side half only: a single Trading-skill roll that can move the wage up from the posted floor, never down. Flagged as a deliberate simplification, not an oversight — the fuller model has real prerequisites this stage doesn't have yet.
- **The seeded hoe's durability (3000, vs. shoes' 200) is deliberately generous**, not balanced. Company equipment *purchasing* — replacing a tool once it breaks — is explicitly Stage 5 (§15), so Stage 3's single tool needs to comfortably outlast a season's worth of shifts (empirically: a season's scripted run wears it by under 1%) rather than strand the job on a mechanic this stage isn't meant to build yet.
- **No separate "storage" container for the farm's grain.** Company-owned items (the hoe, harvested grain) are produced directly into the company's own entity id as their container — the same as a person's inventory — rather than inventing a `<companyId>-storage` suffix like the market stall's stock container needed. The market needed that suffix because a site isn't an entity; a company already is one, so it doesn't.

## 2026-07-14 — The Survival Loop (Stage 2 complete\*, §Stage 2) — \*with two flagged gaps

**What was built:**

- **Needs** (`src/engine/needs/`): `needs` table (hunger/thirst/energy/warmth, 0-100), decaying every tick via a new `Engine.applyNeedsCadence` cadence hook (thirst fastest, per §6). Hitting 0 on any need triggers collapse: interrupts the current action, cancels anything queued behind it, and force-queues a `collapse_recovery` action (always registered — a core mechanic, not world content, same spirit as the automatic nightly audit) that restores all four needs partially on completion.
- **Skills** (`src/engine/skills/`): a generic `skills` table (any skill name, not an enum) with learn-by-doing XP and a steep level curve; only `labor` is used pre-Stage 3 (trade skills arrive with jobs/production). Skill level raises an action's success chance, capped below 100% — "skill affects failure rate," per §13.2, not eliminates it.
- **Gear** (`src/engine/gear/`): one item per slot (`gear` table), `items` gained a `durability` column. Wear reduces durability; hitting 0 unequips and destroys the item (reason `worn_out`) — the mechanism behind "a gear replacement purchase."
- **Market** (`src/engine/market/`): finite seeded `market_listings` per (site, good) — no restocking/price drift, that's Stage 5. `createBuyActionDefinition`/`createSellActionDefinition` generate reusable `ActionDefinition`s: buying sinks the buyer's coin and produces the good (§8.1's "imports," modeled literally — no merchant entity exists yet to receive the coin instead); selling transfers the item into the stall's own storage and grows the listing's quantity (goods conservation — §8.1 rule 1, not a destruction) while fauceting coin to the seller (§8.1's "export purchases").
- **Goods catalog** (`src/engine/goods/catalog.ts`): a code-level table of weight/price/hunger-thirst-restore/warmth/durability per good type — same precedent as action definitions being code until production/recipes (Stage 5) makes goods properly data-driven.
- **Inventory capacity** (`src/engine/inventory/capacity.ts`): a flat per-person carry-weight cap read from the catalog.
- **Action framework extended**: `ActionDefinition.resolve` now also receives an `ActionEffectContext` (db/bus/actorId/tick) — foreshadowed literally in Stage 0's own decision log ("skill checks, later on") — and a new optional `applyOutcome(ctx, outcome)` hook applies the mechanical consequences (produce/consume items, restore needs, spend coin, wear gear) once the outcome is known, keeping `resolve` about "did it work" and `applyOutcome` about "what happens as a result."
- `demoWorld.ts` gained: a `market` site, starting shoes (equipped), and real mechanics for `draw_water`/`rest_bunk`/`rest_rough`/`eat`/`chop_wood` (skill-gated, wears shoes, produces firewood) plus generated buy/sell actions for bread/shoes/cloak/firewood. World seeding is now split into `registerDemoActionTypes` (always runs) and the DB-content seed (idempotent, runs once) — see the bug below for why.
- Headless scenarios (`src/engine/scenarios/stage2.test.ts`): an idle player collapsing on schedule (deterministic ~tick 600, thirst empties fastest), and a scripted worker surviving a multi-day run — gathering, selling, buying, eating, resting, and replacing worn-out shoes — with every consumed item's provenance chain verified end to end (`produced → consumed`).
- HUD needs bars, a worn-gear line, and market price/stock shown on buy/sell buttons (`LocationScreen`).

**Bugs found and fixed along the way (not hypothetical — each was actually hit):**

- **`Engine.advanceTicks` now wraps its whole tick batch in one explicit `BEGIN`/`COMMIT`/`ROLLBACK`**, instead of leaving each tick's writes as their own autocommit statement. sql.js's WASM heap doesn't reclaim per-statement rollback-journal overhead across thousands of individual autocommits, and reliably hit "out of memory" well under 100k ticks without this. Confirmed empirically: 30k ticks unwrapped → OOM; 43.2k wrapped → clean.
- **A real CHECK-constraint bug**: `ActionStatus` grew a `'cancelled'` value for `cancelQueuedActions` (collapse handling), but migration `0002`'s `actions.status` CHECK constraint was never updated to allow it. Every collapse that cancelled a queued action threw a constraint violation — which, left uncaught deep in a mid-transaction call, corrupted the connection badly enough that some *later*, unrelated query would crash with a wasm "memory access out of bounds" (the actual error was real and immediate; what followed was a side effect of continuing to use the connection afterward). Fixed via migration `0006` (SQLite has no `ALTER TABLE` for CHECK constraints, so this recreates the table — fixing `0002` in place would rewrite already-applied history for any existing save).
- **A latent save/reload bug**: action *definitions* are code, held only in the in-memory `ActionRegistry` (a Stage 0 decision), never persisted. `seedDemoWorld`'s single guard clause (`if (engine.getSite('well')) return`) returned *before* registering action types — meaning a reloaded save (or, as it turned out, a rehydrated `Engine` during the investigation below) would get a completely empty registry and throw "Unknown action type" on the first queued action. Split into `registerDemoActionTypes` (unconditional, every fresh `Engine`) and the DB-content seed (still guarded).
- **`ui-api`'s `getActorActions`, and the `Hud`/`ActionQueuePanel`/`useGameClock` calling it every tick refresh**, was the same O(n²) trap in miniature — full unbounded history, re-scanned on every render. Replaced with `getCurrentAction`/`getActiveActions` (targeted queries over just the not-yet-resolved rows), matching the fix already needed for the headless test's own decision loop.
- **`NeedsBar`'s progress fill used `<span>` elements** for the track/fill with a `width` set via inline style — `width` doesn't apply to non-replaced inline elements, so the bars silently rendered as flat lines regardless of the actual need value. Fixed with explicit `display: inline-block`/`block`.

**Known limitation — sql.js WASM memory ceiling (flagged risk, not resolved):**

The Stage 2 exit test asks for a scripted worker to "survive 30 days." The shipped scenario test runs **12 days**, not 30. A realistic mix of action types (buying, eating, resting, gathering, selling — not just one repeated action) reliably exhausts sql.js's WASM heap with a clean `Error: out of memory` somewhere around 30-40k ticks in this Node environment. This was run to ground, not assumed:

- Not row/data volume — tables stay a few thousand tiny rows throughout.
- Not Vitest-specific — a bare `tsx` script crashes at the identical tick.
- Not fixed by requesting more WASM memory at `initSqlJs()` (`TOTAL_MEMORY: 512MB` had zero effect on the crash point).
- Scales with *distinct operation volume*, not raw tick count — simple/repetitive workloads (e.g. one action type, looped) survive 100k+ ticks fine; the same tick count with a varied, realistic action mix fails much earlier.
- Reducing the test script's own decision-query overhead (batching same-type actions before re-deciding) did not move the crash point — the cost is dominated by the engine's own per-action processing (enqueue/start/resolve/`applyOutcome`), not by how the test decides what to queue.

12 days sits comfortably clear of the ceiling and still exercises every required behavior (multiple collapse-free need cycles, at least one full gear wear-and-replace cycle, verified consumption traceability). **This must be root-caused or worked around before Stage 4 (90-day exit test) and Stage 5 (2-year exit test) are attempted** — both will hit it far harder, with multiple NPCs. Candidate directions for that future work: find the actual sql.js/SQLite-wasm allocation bug; a periodic checkpoint/rehydration strategy, which will need real RNG-state persistence to stay deterministic (a naive export/reimport-with-reseed loses determinism across the checkpoint, since the engine's RNG is an in-memory generator, not DB-backed); or moving very-long headless runs off sql.js entirely.

**Not built — carried forward, not silently dropped:**

- **Pre-commit warnings** (§Stage 2's own feature list, §18 risk mitigation "Harsh ≠ opaque: pre-commit warnings") were never implemented. Nothing in `LocationScreen` (or anywhere else) projects whether queuing an action would leave a need critically depleted before the player commits to it — every action queues immediately. This isn't required by the Stage 2 *exit test* wording specifically (which only asks for collapse/recovery and traceability), which is why it didn't block calling the exit test met, but it's a real, named Stage 2 feature that's still missing. Worth picking up early in Stage 3 rather than letting it drift further.

**Decisions:**

- **Collapse cancels the whole queue, not just the current action.** A forced interruption "everything after this no longer applies" reading of §6 — an actor recovering from collapse shouldn't resume a stale multi-step plan afterward. Cancelled queued actions get their own status (`cancelled`) distinct from `interrupted`, since they hold no partial progress to report.
- **Warmth ignores indoor/outdoor location entirely** (decays only by season + equipped cloak warmth). The engine has no location-presence tracking yet (that's Stage 4's population/schedule module) — modeling "cold" as season-plus-clothing is the honest simplification available now, not a permanent design stance.
- **`eat` is a separate action from buying**, deliberately, even though it adds a step. Buying acquires an item; eating consumes it. This is what makes "all consumption traceable" (the exit test's own words) a real, checkable provenance chain instead of an implicit assumption.
- **Selling never destroys the item** — it transfers into the market's own storage container and grows that listing's quantity. Destroying it would have been simpler but wrong: §8.1 rule 1 reserves destruction for spoilage/wear/consumption, and a sold firewood is genuinely available stock for the next buyer (including NPCs, once Stage 4 gives them purchasing behavior).

**Exit-test status (Stage 2, §Stage 2): met, with the flagged gaps above** — idle collapse is deterministic and verified; the scripted-survival scenario is real and passing but covers 12 days instead of 30 for the documented sql.js reason; pre-commit warnings from the stage's feature list weren't built. Stage 3 (First Job, §Stage 3) is next, but the memory-ceiling investigation should get real time before Stage 4.

**Decisions:**
- **TypeScript stays at 6.0.3, not the newly-released 7.0.2.** typescript-eslint@8.64.0's peer range is `>=4.8.4 <6.1.0`; 6.0.3 is already the latest version inside that range (6.1+ doesn't exist yet as a stable release). Upgrading to 7.x would have silently broken all type-aware linting.
- **ESLint pinned to the 9.x line (9.39.5), not the newly-released 10.x.** Every plugin in the stack declares `eslint: ... || ^10.0.0` support except `eslint-plugin-react@7.37.5`, which caps at `^9.7`. "Latest compatible," not just "latest."
- **`eslint-plugin-import-x` needs `eslint-import-resolver-typescript`, not `eslint-import-resolver-node`.** The node resolver produced a wall of "invalid interface loaded as resolver" errors on every `import-x/*` rule — a real resolver/plugin-version mismatch, not a code issue. Swapping resolvers cleared ~80 of the ~81 initial ESLint errors in one change.
- **`db.exec()` returns `[]`, not `[{columns, values: []}]`, when a SELECT matches zero rows** — confirmed empirically (not documented in sql.js's own types), and it's what made `noUncheckedIndexedAccess` errors so pervasive across every `result[0].values[...]` call site. `queryRow`/`queryRows` centralize that distinction once instead of re-deriving it at ~15 call sites.
- **`withOptional()` exists because `Partial<Opt>` alone isn't `exactOptionalPropertyTypes`-clean** — it still carries the `| undefined` from `Opt`'s own inferred value types, which is exactly what that flag rejects at a target like `actorId?: string`. The helper's return type explicitly excludes `undefined` from each optional value, matching what the function actually guarantees at runtime. The alternative (widening every optional prop across the codebase to `T | undefined`) would have defeated the flag's purpose everywhere just to satisfy it in a few places.
- **Prettier excludes `*.md`.** It reformatted `MASTERPLAN.md`'s prose (italics syntax, table padding) on first run — cosmetically harmless, but rewriting the user's authored design doc as a side effect of a code-formatting pass wasn't in scope. Docs keep their own formatting conventions; `format`/`format:check` only touch code and config.
- **`no-base-to-string` fixes use `typeof` narrowing, not `String()` + a suppression.** `SqlValue` includes `Uint8Array`, which the rule won't assume stringifies sanely — even though our schema never actually stores blobs in those columns. Narrowing with `typeof row[n] === 'string'` before parsing is strictly more correct than the blind `String(row[n])` it replaced, not just quieter.

**Verification:** `npm run validate` (typecheck → lint → format:check → test → build) passes clean; all 33 existing tests still pass unchanged (only type-level fixes, no behavior changes); a real browser smoke test confirmed the dev server still boots after the tsconfig/engine-file changes.

## 2026-07-14 — Item provenance + conservation audit (Stage 0 complete, §7.1/§8.1/§16)

**What was built:**
- Migration `0004_inventory_and_audit`: `items` (single-container rule, status lifecycle), `provenance_events` (one item's full history), `wallets` (one balance per entity), `audits` (a row per audit run), plus four running-total counter columns added to `world_meta` (`goods_created`, `goods_destroyed`, `coin_faucet_total`, `coin_sink_total`).
- `src/engine/inventory/` — `items.ts` (`produceItem`/`transferItem`/`destroyItem`/`getItem`/`getProvenanceChain`/`countActiveItems`), `wallet.ts` (`faucetCoin`/`sinkCoin`/`transferCoin`/`getBalance`/`sumWalletBalances`), `counters.ts` (the shared running totals both of the above update).
- `src/engine/audit/conservationAudit.ts` — `runConservationAudit`: compares the counters (what *should* be true) against live `COUNT`/`SUM` queries (what the tables *actually* contain), records a row in `audits`, and emits a `world`-scoped `audit.failed` log event on mismatch.
- `Engine` runs the audit automatically at every in-game midnight (`tick % MINUTES_PER_DAY === 0`) per §4.2's "nightly (conservation audit)" cadence, and exposes `runConservationAudit()` for manual/test invocation.

**Decisions:**
- **The audit compares counters against live tables, not "this audit's totals vs. the prior audit's totals."** Counters (`goods_created` etc.) are incremented only inside `produceItem`/`destroyItem`/`faucetCoin`/`sinkCoin` — so as long as all state changes go through those functions, counters and live tables always agree. If something ever bypasses them (a bug, a bad migration, a raw query), the two diverge and the audit catches it same-day, which is the actual point of §16's "drift = bug." `conservationAudit.test.ts` proves this by deliberately deleting an item row and mutating a wallet balance via raw SQL and confirming the audit flags both.
- **Transfers (`transferItem`, `transferCoin`) don't touch the counters.** Only creation/destruction and faucet/sink change how much exists in total; moving something that already exists between containers/owners is conservation-neutral by construction, matching §8.1 rule 1 ("every transfer transactional and logged" — logged via `provenance_events`/the event bus, not via the conservation counters).
- **`items.status` has no `'destroyed'` value distinct from `'consumed'`/`'spoiled'`/`'worn_out'`** — §8.1 rule 1 says "spoilage and wear are the only destruction," so those three specific reasons *are* the destruction vocabulary; a generic fourth status would just be an unused escape hatch.
- **One wallet per entity, no business ledgers yet.** Business ledgers (§9.3) are a Stage 5 concept with far more fields (revenue, wages, equipment upkeep); building that structure now, before companies exist, would be speculative. `wallets.owner_id` will grow to accept business IDs once that module lands.

**Exit-test status (Stage 0, §Stage 0): all five criteria now met** — 10,000 deterministic headless ticks, a scripted actor completing a queued chain including a failed attempt, save→load→resave byte-identical, the conservation audit passing (and, better than the exit test strictly requires, verified to actually *catch* drift), and a seeded item's provenance chain queryable end to end. Stage 1 (React interface shell, §Stage 1) is next.

## 2026-07-14 — Timed-action framework (Stage 0, §4.3)

**What was built:**
- Migration `0002_entities_and_actions`: an `entities` table (id + name only for now — household, portrait, skills, traits, etc. arrive with the modules that need them) and an `actions` table (one actor has at most one `in_progress` row; `queued` rows behind it are that actor's committed chain).
- `src/engine/actions/` — `types.ts` (`ActionDefinition`, `ActionOutcome`, `QueuedAction`), `registry.ts` (`ActionRegistry`, a type→definition map), `actionQueue.ts` (`enqueueAction`, `processActorActions`, `interruptCurrentAction`, `listActorActions`).
- `Engine` gained `registerActionType`, `createEntity`, `queueAction`, `getActorActions`, `interruptAction`, and a per-tick `processActiveActions` cadence hook that drives every actor with a pending action forward.

**Decisions:**
- **Action *definitions* are code (a `resolve: (rng) => ActionOutcome` function in a registry), not DB data**, unlike goods/recipes which §16 wants data-driven. Actions need arbitrary behavior (skill checks, later on); recipes/goods are the actual data-driven surface once the production module lands. Revisit if action definitions grow complex enough to want hot-reloading or modding (§16) before then.
- **No idle tick between chained actions.** `processActorActions` loops internally so a completion and the next action's start land in the *same* tick call, instead of "resolve this tick, start next tick." An actor with queued work is occupied every tick, matching §4.3 ("occupying the character exclusively") — an idle gap would be a subtle, hard-to-notice inefficiency baked into every job/shift/gather chain in the game.
- **`interruptCurrentAction` only acts on an `in_progress` action**, not a `queued` one (cancelling something that hasn't started is a different, not-yet-built operation). Progress is stored as elapsed ticks, not a rounded fraction, so exact interruption timing survives a save/reload.
- **`processActiveActions` orders by `actor_id ASC` explicitly** rather than trusting `SELECT DISTINCT`'s incidental row order — needed since RNG draws happen during resolution, and draw order must be reproducible for the same seed.

**Exit-test progress:** "scripted actor completes a queued action chain including a failed attempt" (Stage 0 exit test) ✅ — `src/engine/actions/actionQueue.test.ts` queues chop→fail→chop for one actor and asserts status, timing, and log output. Still open: grid-coordinate world model, conservation audit, provenance recording.

## 2026-07-14 — Project scaffold (Stage 0, partial)

**What was built:**
- Vite + React 19 + TypeScript shell (`npm run dev` / `npm run build`), Vitest for headless engine tests (`npm test`).
- `src/engine/` — pure TypeScript, zero React/DOM dependency, per §4.2:
  - `db/` — sql.js bootstrap split into `sqlite.ts` (shared wrapper), `sqlite.node.ts` (Node/tsx/Vitest, locates the wasm binary via `require.resolve`), `sqlite.browser.ts` (Vite app, fetches `/sql-wasm.wasm` from `public/`).
  - `db/migrations/` + `migrationRunner.ts` — ordered migrations, tracked in a `schema_migrations` table, applied idempotently. Schema v1 so far only has `world_meta` and `event_log` (§16's full data model — entities, items/provenance, actions, inventories, etc. — lands incrementally, one module per session per §19).
  - `eventBus.ts` — the event bus modules communicate through (§4.2's "DB + event bus only" rule).
  - `rng.ts` — seeded deterministic RNG (mulberry32 + FNV-1a string hashing) so "same DB + same seed = same result" (§4.2) is actually testable.
  - `time/clock.ts` — tick → minute/day/season/year derivation (§4.3 calendar).
  - `logs/logger.ts` — subscribes to the event bus, writes to `event_log`, exposes scoped queries (personal/business/settlement/world, §14.3).
  - `engine.ts` — the `Engine` class: bootstrap (migrate + seed world_meta), `advanceTicks`, `export`/save bytes, log queries. The tick loop itself is currently a no-op counter increment; module cadence hooks (needs, actions, market, ...) attach here as each module is built.
  - `ui-api/` — the narrow engine↔UI surface (`createUiApi`) React is meant to consume instead of reaching into `Engine`/`db` directly.
  - `headless-runner.ts` — `npm run sim:headless [seed]`, runs 10,000 ticks and prints tick/calendar/world-log, per the Stage 0 exit test.
- Directories created (currently empty, populated as each module is built): `needs`, `actions`, `jobs`, `production`, `inventory`, `market`, `households`, `companies`, `construction`, `housing`, `transport`, `population`.

**Decisions:**
- **`schema_migrations` stores no timestamp.** A wall-clock `applied_at` column made two identical-seed runs export different DB bytes, which directly violates the §4.2 determinism invariant. Migration provenance already lives in git history; the DB only needs to know *which* migrations have run, not *when*.
- **sql.js bootstrap is split by platform (`sqlite.node.ts` / `sqlite.browser.ts`), not unified with a runtime check.** Locating the wasm binary is inherently platform-specific (Node `require.resolve` vs. a `fetch`-able `public/` path); trying to unify it behind one `typeof window` branch risks bundlers pulling `node:module`/`node:path` into the browser chunk. The actual engine logic (schema, migrations, tick loop) is unaffected and shared.
- **`tsconfig.app.json` includes `"node"` in `types`**, even though it's the browser app's tsconfig. `tsc -b`'s `include: ["src"]` pulls in `sqlite.node.ts`/`headless-runner.ts`/tests transitively regardless of any `exclude`, so they need Node's ambient types to type-check; this doesn't add any actual Node runtime dependency to the browser bundle.
- **Calendar model:** 30 days/season × 4 seasons = 120-day year, tick = 1 in-game minute (§4.3). No leap logic, no partial seasons — reassess if the balance harness (§17) needs finer control.
- **`sqlite.browser.ts` imports `sql.js/dist/sql-wasm.js` directly, not the bare `sql.js` specifier.** Found by actually driving the app in a browser (Playwright): Vite's `browser` export condition resolves the bare specifier to `dist/sql-wasm-browser.js`, whose companion binary is `sql-wasm-browser.wasm` — a second wasm file to keep in sync in `public/`, and one that silently 404s into the SPA's `index.html` fallback if you only copy `sql-wasm.wasm`. Pinning both platforms (`sqlite.node.ts` and `sqlite.browser.ts`) to the same explicit build keeps there being exactly one wasm file (`public/sql-wasm.wasm`) to remember to update if `sql.js` is upgraded. `vite build` prints harmless "node:fs/node:crypto externalized for browser compatibility" warnings for this same file — those paths are guarded by Node-only feature checks that never execute in the browser bundle; confirmed clean (no runtime errors, tick/log UI works) via an actual Playwright-driven browser run.

**Exit-test status (Stage 0, §Stage 0):** 10,000 deterministic headless ticks ✅, save→load→resave identical bytes ✅ (both covered by `src/engine/engine.test.ts`). Not yet done: timed-action framework with failure outcomes, grid-coordinate world model, conservation audit, provenance recording — these are separate module sessions per §19.
